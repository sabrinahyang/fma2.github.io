Pairing & Giving Feedback

Jun 15th, 2014

Truth be told, pairing initially felt awkward. It meant I had to meet with a stranger via video chat and accomplish a task–which required abilities that I was initially not completely confident I had–in a short period of time. To add to the awkwardness, I was nervous I would not meet the expectations set in Pairing is Caring. Notably, I did feel like I understood the purpose and benefits of pairing in theory.

During the second guided pairing session, it finally clicked–I found I was finally as comfortable with pair programming in practice as I was positive about it in theory. In retrospect, I realize I had not been confident in my technical abilities and was nervous I would be more burdensome than useful. For the second GPS, a guide served as my pair, and he used simple questions when I was stuck to help me me move forward. The questions did not hand me any answers; rather, they made me think about the problem differently, which led me to quickly solve it. Perhaps it was his experience with pairing that made him such a successful guide; either way, it allowed me to experience the power of pairing.

Eventually, I grew to greatly appreciate pairing, even with the occasional awkwardness of having to use video chat instead of the ideal of meeting in person. I try to avoid using cliches (sometimes because I don’t understand them as an English Language learner!), but pairing definitely showed that “two heads are better than one.” It was rewarding in so many ways, including the chance to talk out code and find I was on the same page as my pair and then eventually to solve the problem in less time than it would have taken me on my own. I was only occasionally frustrated with myself. When I saw that I was not on the same page as my pair, I felt I needed to communicate more clearly and sometimes was not as comfortable with relevant vocabulary to do so.

For my current full-time job, there are few formal (or informal) protocols in place for feedback. I wish there were more systems, so I unsurprisingly was thrilled to see that not only was feedback expected at DBC but there were also specific guidelines that included a rubric that asked for feedback that was specific, actionable, and kind; timelines to follow; and a check for good feedback through metafeedback.

It was helpful to read feedback; it often confirmed what I thought I needed to improve but also provided other ways to grow I had not considered. Occasionally, the feedback rehashed the success of the pairing session, so it was fun to relive it through written word. Further, it served to keep me accountable–I knew from the beginning that I would l likely need to spend extra time on technical aspects (as a complete newbie to web development), so, in preparation for pairing sessions, I would do this extra work to ensure there was equity of effort during the session. As a result of feedback, I also reflected on my communication style/s and considered what to remember to do or say during a pairing session.

I actually found writing feedback to be the most difficult part of the process. Notably, I think it is because, while I realize feedback is innately subjective, I felt this need to make it as objective as possible, especially considering the guidelines and expectation to make it specific, actionable, and kind. With that in mind, reviewing others feedback provided valuable examples of how to write good (and occasionally not so good) feedback.

Conclusively, while pairing via Google Hangouts is fine and receiving feedback on these sessions is great, I am mainly looking forward to being able to engage in this productive process in person when Phase 1 begins!

Test-driven Development

Jun 14th, 2014

Test-driven development (TDD) is “a software development process that relies on the repetition of a very short development cycle”1.

A graphical representation of the development cycle, using a basic flowchart1

Generally speaking, the cycle includes the following steps:

  • (1) Decide what the code will do
  • (2) Write a test that will pass when the code runs
  • (3) Run the test, see it fail
  • (4) Write the code–just enough production code to fulfill the written test
  • (5) Run the test, see it pass
  • (6) Repeat the above process for every couple of lines of code 2

    • Some refer to the steps as RED, GREEN, and REFACTOR.

  • 1) Understand the requirements of the story, work item, or feature that you are working on.
  • 2) Red: Create a test and make it fail–it is expected for it to fail, as it ensures your code is working.
  • 3) Green: Make the test pass by any means necessary. Write the production code to make the test pass. Keep it simple.
  • 4)Refactor: Change the code to remove duplication in your project and to improve the design while ensuring that all tests still pass. Repeat the cycle. Each cycle should be very short, and a typical hour should contain many Red/Green/Refactor cycles3.

    • Check out this video for more on this process:

    As the video describes, TDD encourages design of testable code, which means better planning leading to better results. Amongst many uses, writing testable code through TDD forces the developer to break things down (i.e. code is modular), ensures code is not too tightly woven (referred to as decoupled design), and includes methods with limited scope.

    Benefits of TDD

  • - TDD enables the developer to take small steps when writing software. Many find this practice to be more productive than attempting to code in large steps. Further, it is easier to find and fix defects in code4.
  • - Each test provides feedback to the developer that each component is working.
  • - The tests also provide documentation to the developer
  • - After the test passes and the code is refactored, “it is clear that the code is finished, and the developer can move on to a new test” 3
  • - Test-driven development forces critical analysis and design because the developer cannot create the production code without truly understanding what the desired result should be and how to test it 3.
  • - Reduces debugging time.
  • - Overall, advocates of TDD feel it leads to better design in less time.

    • Objections to TDD

  • - To be able to write better code in less time, one needs to get used to TDD before he/she gets faster.

  • - The basic steps are easy to learn but the mindset takes a while to sink in. And, before it sinks in, “TDD will likely seem clumsy, slow, and awkward”5. It may take 2 to 3 months of full-time TDD use to adapt.

  • - It promotes microdesign over macrodesign and can be hard to apply in practice6. Here is an interesting blog post on these points.

    • Affirmation & Stereotype Threat

    Jun 8th, 2014

    In the article 15-minute writing exercise closes the gender gap in university-level physics, the author describes Akira Miyake’s study to close the gap between male and female performance. In this study, students in a physics class participated in a simple writing exercise that asked students to think about the things that are important to them; pick two or three of these values and write a few sentences aobut why they are important. Miyake believes this “value-affirming exercise has two benefits: it break the vicious cycle of stereotype threat, but it also sets up a positive cycle too”1. This exercise is very similar to our previous blog post on “Exploring your Values”.

    When I think about the values described in my previous blog post, I first feel protective, then grounded. I feel protective because the values are important to me and, when revealed publicly (a website/blog post can be public), inspire a sense of vulnerability. I subsequently feel grounded because, despite various day-to-day external demands, I am able to understand myself by reconnecting with these values–they form my identity.

    I think affirming my values can help me mediate stereotype threat. Steele speaks of the value of coaching people to make an identity less salient2 in response to stereotype threat, and discussing my values likely begins this process. Because my values form my identity, I make their importance known when I affirm them; I will begin to be more conscious of their relevancy in/to stereotype threat situations. Further, because they are available for my cohort-mates to view on my blog, they are also a useful reference point when discussing stereotype threat experiences with them.

    When a situation gets me down, I usually take a step back to reflect and reassess. I consider causes for the negative feelings and how I can resolve it–whether it be a personal mindset change or a conversation with someone else. At DBC, amongst other moments of self-doubt, I anticipate there will be moments where I question my abilities or question how well I am communicating or have communicated with my peers. To counter, I think I will continue to use reflection and reassessment. Specifically, I will revisit reasons I chose DBC to connect with what I value in the program and handle the situation accordingly.

    • Optimizing Sql Queries

    Jun 7th, 2014

    This post will discuss ways to optimize SQL queries. I will limit it to SQL query optimization options within my current scope of understanding of SQL syntax in an effort to apply the tips as soon as possible. However, I should note that, when researching the topic, I found various other SQL query optimization suggestions.

    SQL query optimization is important for better performance. A query is a request for information from a database. Database structures are complex, so it is not surprising that “the needed data for a query can be collected from a database by accessing it in different ways, through different data-structures, and in different orders”1–this is performed through SQL statements. What can change with different SQL statements is processing time. Query optimization is meant to “find the way to process a given query in minimum time”1. Why optimize SQL queries? ServerWatch argues, “Time is money and people don’t like to wait so programs are expected to be fast”2.

    Here are two approaches to optimize your SQL queries with a few examples of each in practice–

    APPROACH #1: ONLY QUERY WHAT YOU REALLY NEED

    Use the actual column names in SELECT statements instead of ‘*’. “This is, hands-down, the biggest, baddest SQL optimization violation out there. If you considered “SELECT *” to be off limits in your applications, I bet that would clear up a sizable chunk of the SQL problems that applications have”3

    BAD

    SELECT *
    FROM users;

    GOOD

    SELECT id, first_name, last_name, age
    FROM users;

    Be careful with conditions in the WHERE clause. Write it according to what you need/what you want.

    BAD

    SELECT id, first name, last_name, age
    FROM users
    WHERE age != 10;

    GOOD

    SELECT id, first name, last_name, age
    FROM users
    WHERE age > 10;

    BAD

    SELECT id, first_name, age
    FROM student_details
    WHERE SUBSTR(first_name,1,3) = ‘Cha’;

    GOOD

    SELECT id, first_name, age
    FROM users
    WHERE first_name LIKE ‘Chan%’;

    Use the HAVING clause appropriately. It is used to filter the rows after allthe rows are selected; if you do not need to do this, then you do not need a HAVING clause 4

    BAD

    SELECT subject, count(subject)
    FROM student_details
    GROUP BY subject
    HAVING subject!= ‘Vancouver’ AND subject!= ‘Toronto’;

    GOOD

    SELECT subject, count(subject) FROM student_details
    WHERE subject != ‘Science’
    AND subject != ‘Maths’
    GROUP BY subject;

    APPROACH #2: CONSIDER HOW DIFFERENT OPERATORS WORK/WHAT OPERATORS ARE APPROPRIATE

    Use operator EXISTS, IN,and table joins appropriately in your query. “Usually, IN has the slowest performance. IN is efficient when most of the filter criteria is in the sub-query. EXISTS is efficient when most of the filter criteria is in the main query.”4

    BAD

    SELECT *
    FROM product p
    WHERE product_id IN (SELECT product_id FROM order_items)

    GOOD

    SELECT *
    FROM product p
    WHERE EXISTS (SELECT * from order_items o
    WHERE o.product_id = p.product_id)

    Try not to use the NOT operator. It is much faster to search for an exact match using the LIKE, IN, EXIST or = symbol operator (positive operators) instead of NOT LIKE, NOT IN, NOT EXIST or != symbol (negative operators). A negative operator will “cause the search to find every single row” whereas the positive operator stops as soon as the result is found.

    BAD

    SELECT name
    FROM users
    WHERE name NOT LIKE “F%”;

    GOOD

    SELECT name
    FROM users
    WHERE name LIKE “M%”;

    Minimize the number of subqueries in your main query.

    BAD

    SELECT name
    FROM employee
    WHERE salary = (SELECT MAX(salary) FROM employee_details)
    AND age = (SELECT MAX(age) FROM employee_details)
    AND emp_dept = ‘Electronics’;

    GOOD

    SELECT name
    FROM employee
    WHERE (salary, age ) = (SELECT MAX (salary), MAX (age)
    FROM employee_details)
    AND dept = ‘Electronics’;

    Use EXIST instead of COUNT when trying to determine whether particular data exists. The EXIST operator will stop once it finds the first record while the COUNT operator will search for all existing records on the table.

    BAD

    SELECT column
    FROM table
    WHERE COUNT(column) > 0

    GOOD

    SELECT column
    FROM table
    WHERE EXISTS(SELECT * FROM customer
    WHERE grade=3)

    Hopefully, these tips will help you get started in optimizing your SQL queries! For more, check out these resources:

    • Stereotype Threat

    Jun 1st, 2014

    Stereotype threat “refers to being at risk of confirming, as self-characteristic, a negative stereotype about one’s group”1. While I am familiar with some of the research on stereotype threat, I had not previously seen any of Steele’s talks. Notably, even more than any article I have read on the topic, his presentations made me realize how prevalent the issue is–how it is not limited to a specific community or space (academic, personal, work, etc.).

    An individual may have numerous social identities, including but not limited to age, sex, religious affiliation, race, occupation, etc; Steele argued that it is possible that no two people have the same set of social identities2. The social identities that one “can’t put down” are called identity contingencies. If the social identity requires one to deal with something in a particular situation as a result of the identity, then it is likely that social identity will become more important to that person–this is an identity contingency.

    As an example, Steele describes when he first realized he was black. Similarly, while I recall identifying as Muslim early in life, I more clearly saw myself as different from many of my peers around Christmas each year. I attended an elementary school with mostly white students in a suburb in California; I did not find it notable that I did not celebrate Christmas until I was not able to relate with my peers, serving only as a starting point for numerous other moments where I felt like an outsider. Interestingly, I related with Muslim peers outside of school with ease, a discernible contrast from the othering I felt at school.

    While my example does not explicitly speak to stereotype threat, I feel it is necessary to begin a conversation on the issue with an understanding of social identities and identity contingencies, as both are relevant to the source of and perhaps the response to stereotype threat. Steele argues that the most psychologically impactful identity contingencies are those that in some way threaten the individual–stereotype threat is a subtle identity contingency. He cites Amin Maalouf from In the Name of Identity: Violence and the Need to Belong: “People often see themselves in terms of whichever one of their allegiances (identities) is under attack…whether he accepts or conceals it, proclaims it discretely or flaunts it, it is with that allegiance that the person concerned identifies”3.

    With that said, I have felt stereotype threat in numerous situations. For one, while I love traveling, I hate airports and airport security. I identify as South Asian, Muslim, and American (born and raised), and I have been stopped for extensive searches at airports more times than I can count. What’s more, on all international flights back to the U.S. save for one, I have been subject to numerous “random” searches, sometimes at all available checkpoints. Once, while departing for the US from a country in South America, I was stopped 7 times; at some point, it stopped feeling random.

    In this particular instance, I could not help but get angry; at the last of the 7 searches, I lectured the safety agent (sidebar: it worked – they did not search me as thoroughly as they did the person before me). Unfortunately, I also could not help but feel incredible guilt after the fact. I did not like that I had possibly validated stereotypes about the Muslim and/or South Asian community; I know had reason to feel aggressive but still could not stop feeling guilty for misrepresenting Muslims or South Asians. I feel this burden every day–at work, on the street, with friends–especially in light of NYPD surveillance of the Muslim community and related post-9/11 Islamophobia. It means I have to temper speaking up against injustices with a desire to make sure my actions or words adequately represent the Muslim and/or South Asian community while also not making the situation any worse for them. Lots of pressure, I know.

    Beyond the Muslim and South Asian social identities, presently, other significant identity contingencies in which I feel or recently felt stereotype threat include: age at work (I’m the youngest by 15-20 years and was hired at age 25 in a career, teachers’ union representative, that many do not associate with youth); identification as an activist (this video sums up the stereotype threat I feel in this respect: http://www.youtube.com/watch?v=aKucPh9xHtM&feature=youtu.be; and occupation while a student at an education graduate school (I had trouble participating in classes for fear the points I would make would be overshadowed by how others perceived teachers’ unions).

    To respond to stereotype threat, I think it is important to consider research on stereotype threat. Specifically, Steele speaks of the value of coaching people to make an identity more or less salient. Research “demonstrates that within a stereotyped group, some members may be more vulnerable to its negative consequences than others; factors such as the strength of one’s group identification or domain identification have been shown to be related to ones’ subsequent vulnerability to stereotype threat”1. In any particular situation, group, or community–including a learning community like DBC–it would be valuable to recognize who is most vulnerable to stereotype threat as well as the cues that cause it. Our cultural blog posts on stereotype threat could be a great resource for this. Acknowledging and understanding that these threats may exist is a first step. Afterward, there should be a collective effort to ensure those who identify with particular contingencies are not feeling threatened due this association. These efforts can begin with language – as a blatant example, it is misguided to tell a class that the most intelligent will succeed on a particular learning task.

    In the same vein, Steele describes the value of engaging leadership and using normative influence. Leaders set norms for how organizations and institutions function; they can make a space more or less identity safe. DBC seems to have a strong culture of feedback; it is not just a practice in which Boots and instructors naturally engage, it is also an organizational policy and expectation. DBC leadership has explicitly set this norm. Within it, there could–and should–be explicit attention given to making feedback identity safe. “Making feedback identity safe” involves understanding cues and mechanisms behind stereotype threat and considering how one should modify how they provide feedback, what words they specifically use.

    To make the environment around me more welcoming, it is important for me to be conscientious and self-reflective. I hope to be (and hope that I have been in the past) purposeful with how I communicate, reflective on how it may come across, and responsive when I find I should correct myself or if I see I am able to help someone.

    Sources:

    • 2) Stereotype Threat at Work

    • 3) Video: Identity and Stereotype Threat


    Blocs, Procs, and Lambdas

    May 31st, 2014

    This week’s technical blog covers blocks, procs, and lambdas. I opted to research blocks, procs, and lambdas because, while I recall using them through Code Academy, I was not able to connect them to what I have learned in the past few weeks without some Googling.i

    Blocks, procs, and lambdas are all examples of closures. Closures are functions or references to functions that are bound to the scope in which they were created. Closures meet the following criteria: can be passed around as a value; can be executed on demand by anyone who has that value; and, can refer to variables from the context in which it was created (even if those variables are no longer in scope) (source: http://www.technicalecstasy.net/closures-in-ruby-blocks-procs-and-lambdas/). They are helpful for: refactoring, customization, iterating across collections, managing resources, and enforcing policy 1.i

    Blocks are the most commonly used closure. They are also amongst the few exceptions to the “everything is an object” rule in Ruby. They are considered nameless functions, and they use curly braces {..}or do…endsyntax. They can’t be saved to a variable - so they don’t have the powers of a Ruby object.i

    Below are examples of blocks.

    In Example 1, the block is just a function that takes a single argument x and prints that argument to the screen2:i Example 1 i

    In Example 2, the #collect! method is used on an Array with a block of code. Within the method, the block takes each variable in the array and squares it.i Example 2 i

    Though blocks are useful and syntactically simple, their namelessness presents a problem when one wants to have many different blocks to be used multiple times. To do this, the block would need to be saved as a variable. This is where Procs come in! To save a block to a variable, you have to make it a Proc! i

    “Proc” is short for procedure. Basically, procs are saved blocks. Because they are saved, they are objects that can be passed around. Procs can help make code DRYer. And, as previously alluded, they allow one to write a function once and then pass it to lots of different functions. A Proc is made by using Proc.new and passing a block, by Proc.new {|x| x}, or by using the proc keyword, prox {|x| x}3.

    Notable aspects of procs to remember:i

    • A return from inside exits completely out of the method enclosing the Proc.3
    • A Proc doesn’t care about the arguments passed. If you define a Proc with two parameters, and you pass only 1, or possibly 3, it keeps on trucking. In the case of 1 argument, the second parameter will have the value nil. If you pass extra arguments, they will be ignored and lost.3

    Lambdas, on the other hand, are a more strict form of proc, though they are used almost identically like procs. Specifically, lambdas “check the number of arguments passed to them, whereas procs do not. Lambdas return differently than do procs”2. Their main differences include: lambdas check the number of arguments received and return an Argument Error if they do not match; when a proc encounters a return statement in its execution, it halts the method and returns the provided value, and lambda returns the value fo teh method and allows it to continue4.

A class is the factory that builds an object. To create a class, define the characteristics of the class through methods.     Instance Variable

    An instance variable begins with @. Instance variables can be referenced by any method of a particular class. They are bound to the instance of the particular class. Instance variables are initialized through the instance method initialize.
